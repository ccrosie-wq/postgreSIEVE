\begin{abstract}
    In an world defined by rapid data growth, the demand for near-instantaneous data access has become ubiquitous. 
    To meet these latency requirements, modern systems extensively employ caching mechanisms to store frequently accessed data, thereby mitigating costly I/O operations. 
    However, caches are finite have limited memory, that is where cache evection policies come into play. 
    Eviction policies help determine when data in a cache is no longer needed and removes it creating room for new data. 
    There are many popular ways of achieving this through different algorithms, some of the most common algorithms are Least Recently Used (LRU), Least Frequently Used (LFU), 
    First In First Out (FIFO), and many others. These algorithms can be modified to the systems needs to improve on the native behavior of the policy. 
    Today many database systems leverage these policies to minimize high-latency disk operations and maintain "hot" data in memory. 
    PostgreSQL is such one that naively implements the Clock Sweep cache eviction policy. 
    Clock Sweep is a take on the LRU policy and its designed to minimize the overhead of maintaining an ordered list of recently used pages. 
    For our project we attempt to implement SIEVE, an eviction policy originally designed as a better LRU but for web caches into PostgreSQL. 
    SIEVE is advantageous due to its lock-less design and its ability to be integrated into existing cache architectures. 
    Whilst SIEVE was originally made for the web, there could be possible use for it in database systems such as PostgreSQL. 
    Our experiments benchmark Sieve against other eviction policies using the TPC-C workload and PG-Bench. [Insert quick synopsis of test results - RYAN]
\end{abstract}

\section{Architecture}
% =============================================================================
We use a new eviction policy framework which routes high level hit, insert, and
invalidate calls from the buffer manager through to methods specific to each
eviction strategy. This approach was chosen with the goal of allowing the eviction
algorithm to be chosen at runtime rather than requiring recompilation of the source
code, or switching between different versions of the project.

\subsection{SIEVE Implementation}

Our SIEVE implementation uses two arrays, SieveNext and SievePrev, to track page
positions and emulate a linked list, along with two \emph{head} and \emph{hand}
pointers to manage the SIEVE algorithm. Each array is sized according to the
number of buffers configured, where the Nth element of SieveNext and SievePrev
point to the head and tail respectively. During an insertion or invalidation,
the \emph{head} pointer is updated, and SieveNext and SievePrev values are
updated to reflect the new ordering of the list.
